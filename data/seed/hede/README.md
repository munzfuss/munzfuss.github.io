# Hede seed data — Danish royal-mint coins from danskmoent.dk

Coin entries lifted from the parsed Hede 1971 catalogue cache
(`scripts/cache/hede/*.json`) shaped as Coin-schema records and
**auto-merged into `data/locations/<location_id>.yml` at build time**
by `scripts/build.py::_merge_seeds_into_raw`.

Each file under `data/seed/<source>/<location_id>.yml` whose stem
matches a location's id is treated as a seed side-car for that
location: its `coins:` list is appended to the location's own coins
before schema validation. The seed file is the canonical
intermediate — filtering by year + mint happens at seed-generation
time, not in the build, so the file the build reads is already
scoped to what should ship.

The build is silent about merges that resolve to zero coins; when it
finds a non-empty seed, it prints one line per location summarising
the source + coin count (e.g. `🌱 denmark: merged 373 seed coins
(373 from hede)`).

## Files

| File | Coins | Scope | Mints |
|---|---:|---|---|
| `denmark.yml` | 373 | `year_first ≤ 1914` | København / Frederiksborg / Helsingør / Kbh+Altona |

The seed's `scope_year_to` header field declares the upper-bound
year cap applied at generation time (set via the `--year-to` flag,
default 1914 — the project's scope upper bound per CLAUDE.md).

Generated by `scripts/maintenance/build_hede_denmark_seed.py` from
the parsed Hede cache. Re-run the script any time the parser
(`scripts/parse_hede.py`) gains new fields or the cache grows.

## Coverage

* **Years**: 1541 (Christian III) → 1914 cap (Reichsgoldmünzfuß-era
  Kroner). Modern post-1914 issues (Christian X 1915-1947 jubilees,
  Frederik IX 1947-1968 commemoratives) are dropped at generation
  time by the `--year-to 1914` filter — project scope per CLAUDE.md.
* **Rulers**: Christian III–X, Frederik II–IX (every Danish king
  from the post-Reformation period onward except where Hede 1971
  lacks coverage — early Frederik III's two-Christian-IV-overlap
  years and minor royal-portrait variants).
* **Categories**: kurant / scheide / tarif (Christian IV
  Krone / Corona Danica).
* **Metals**: silver / gold / billon (inferred from nominal-tokens +
  fineness — billon = silver fineness < 0.30).

## Seed conventions

Mirrors the existing `seed_unsorted` bulk-ucoin block at the bottom
of `data/locations/denmark.yml` (and the `seed_unsorted/A` phase
is bumped to `year_to: 1914` to fit the seed's year cap):

* **id**: `dk-hede-<volume><number>` — e.g. `dk-hede-c5h120` is the
  1683 Speciedaler under Christian V, Hede 120.
* **fuss / phase**: `seed_unsorted` / `A` — placeholder bucket. On
  promotion, reclassify each coin into the actual Müntzfuß window
  (use the `Marken fin udbragt til` value from
  `scripts/cache/hede/<basename>.json` as the Müntzfuß indicator —
  9 dlr ≈ 9-Fuß, 9.288 spdlr ≈ 9¼-Fuß, 11.333 rd ≈ 11⅓-Fuß, etc.).
* **All `*_verified` flags `false`** — Hede-seed data needs a
  per-coin sanity check (specimen variance, sub-variant fineness,
  per-year edge cases) before any flag flips to `true`.
* **Source**: `type: literature`, URL pointing at the cached
  `danskmoent.dk/{chr,fr}/<basename>.htm` page.
* **`issuing_entity: danish_realm`** — adjust to `gesamtstaat` on
  promotion when the coin is dated to the Danish-Norwegian Helstat
  period (1660–1814) and was minted at Kopenhagen for joint
  circulation.

## Promotion path

The seed already renders on the live denmark page as part of the
`seed_unsorted` bucket — no manual paste is needed to get the data
visible. Promotion is about moving each entry OUT of the catch-all
bucket into its proper Müntzfuß window inside
`data/locations/denmark.yml`:

1. **Choose an entry to promote** from the seed file. Cut the YAML
   block.
2. **Re-bucket fuss + phase.** Drop `seed_unsorted` / `A`; pick the
   Müntzfuß and phase that match the coin's
   `Marken fin udbragt til` (from the parsed Hede cache) and year.
   Add a new phase to `denmark.yml` if the period isn't covered.
3. **Paste into `data/locations/denmark.yml`** under the chosen
   fuss + phase's coin block, alphabetically by year.
4. **Cross-check against the Hede page.** Open the URL in
   `sources[0]` and confirm `nominal`, `weight_rough_g`, `fineness`,
   `year_label`. The seed is verbatim from danskmoent.dk; the page
   may carry sub-variants the seed flattened.
5. **Add a `bruun_*` citation** when Bruun Part II (the 2025
   Stack's Bowers catalogue) lists a specimen — `bruun_collection_id`
   primary, `bruun_part` / `bruun_lot_no` / `bruun_page` for
   look-up. The Bruun cache (`scripts/cache/bruun/`) is the source.
6. **Flip `*_verified: true`** field-by-field as each value is
   confirmed by an acceptable source per CLAUDE.md §5 (coin
   inscription, museum catalogue, Hede + Bruun cross-attestation,
   Sieg, Numista entry with direct data, etc.).
7. **Run** `python scripts/build.py --validate-only` after each
   batch of integrations to catch chronology mismatches, missing
   phase definitions, or schema drift.

There is no need to delete the entry from the seed file: re-run
`python scripts/maintenance/build_hede_denmark_seed.py` to
regenerate the seed from the parsed cache (a future enhancement
will skip Hede entries whose IDs already exist in the location;
for now they collide on schema-validation as a hard reminder to
remove the seed entry once it's promoted).

## Regenerate

```bash
python scripts/parse_hede.py --force          # if parser changed
python scripts/maintenance/build_hede_denmark_seed.py
```

The seed builder is idempotent — overwrites `denmark.yml` from the
current state of the parsed cache. Re-run safe.

## Excluded from the Denmark seed

Filtered out at build time (would belong in a separate seed for the
relevant location):

* **Schleswig-Holstein mints**: Glückstadt, Altona, Flensborg,
  Haderslev, Rethwitsch, Rendsborg — destined for a future
  `schleswig_holstein.yml` seed.
* **Skåne / Norway**: Christiansstad, Altona+Kongsberg —
  region-ambiguous, defer.
* **No mint extracted** — pages where the H1 parser couldn't
  disambiguate the mint segment (typically multi-section catalogue
  pages or overview pages).
* **Multi-nominal pages with unparseable nominal-list** — pages
  whose H1 carries a comma-list (e.g. «1, 2, 3 og 4 Speciedaler»)
  that the regex couldn't deterministically pair to per-sub-Hede
  denominations.
* **No specs block** — very small Skilling / Hvid entries where
  danskmoent.dk doesn't publish Brutto / Finhed data.
