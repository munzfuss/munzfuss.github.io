## Catalog-index sort + range-collapse (later same day)

User: В«Schou-РґС–Р°РїР°Р·РѕРЅРё (в‰Ґ3 РїРѕСЃРїС–Р»СЊ в†’ range), С‚Р°РєРѕР¶ Р·СЂРѕР±Рё СЃРѕСЂС‚СѓРІР°РЅРЅСЏ РІСЃС–С…
С–РЅРґРµРєСЃС–РІВ» with the messy example `Schou# 26, 28-29, 23-24, 25-26, 36, 9, 10`
в†’ wants sorted + correctly collapsed.

Implemented in `compute._compute_catalog_groups` (`94d6213`): a nested
`collapse_runs()` replaces the old per-value `sorted(set(...), key=value_key)`.
It expands every value to its integer members FIRST вЂ” so existing ranges and
overlapping/adjacent inputs merge (`23-24`+`25-26`+`26` в†’ the int set
{23,24,25,26} в†’ `23-26`), not a messy unsorted overlap. Runs of в‰Ґ3 consecutive
ints collapse to `min-max`; 1-2 runs stay individual numbers; non-integer
tokens (letter sub-variants `93A`, dotted `77.1`, register-qualified) never
collapse and keep the existing `value_key` natural order (parent before
sub-variant). Applies to ALL index groups, not just Schou.

User example в†’ `9, 10, 23-26, 28, 29, 36`. Verified on KM-42 (dk-tid-163034):
`Schou# 21, 24, 25, 28, 32-36, 40, 42, 46-48, 51, 52, 56-59, 61, 62, 64, 68,
71-76, 83, 89, 90, 93, 95, 97-99` (was a 38-number flat unsorted-ish list).
2-element runs render as two numbers per the stated в‰Ґ3 rule вЂ” one-line
threshold flip if в‰Ґ2 is wanted later.
