## The sticky-absorb problem (re-validate)

The cross-source MERGE stage (`merge_seeds_cross_source`) re-runs union-find
from scratch each time, so the shipped nominal discriminator separates
wrong-nominal coins there correctly. But the ABSORB stage
(`absorb_seeds_into_final_v2`) is additive + STICKY: once a unified entry is
in a foundation's `composed_of`, no later run re-validates it. So mis-
groupings that predate the discriminator (and V1-bootstrap composed_of)
survive forever. KM 42 «8 Skilling» (`dk-tid-163034`) carried «1 Denning»
(0.44 g, `unified-kmk-137199`) + «4 Skilling lybsk» (1.822 g,
`unified-kmk-294714`) + two «6 Skilling» — verified via direct
`match_pair(member, foundation)` returning `no_match`.

### Why a NEW pass, not the existing over-merge purge

The over-merge purge (§9.4) handles a curator-reviewed allowlist of
DIFFERENT-base-KM evictions, requires a clean foundation TWIN (resets the
host's accumulated fields to it), and 6 of my 12 KM-42-class hosts are
curator-synthetic ids with NO twin. So I built a separate criterion-driven
`_revalidate_composed_of`.

### The two design traps caught

1. **Weight-tier would false-drop specimens.** Running the FULL `match_pair`
   over existing members gave 38 drops — but 24 were SAME-nominal, dropped
   only by the weight-tier-1 disambiguator (Δ>5 %). That tier is for SPLITTING
   different coins at merge time; re-applied to membership it kills legitimate
   specimen variance (a worn / off-standard piece). Fix: re-validate on
   IDENTITY only — genuine nominal differ + no type-level catalogue tie. Drops
   12, all genuinely different nominals; royal_holstein + holstein_schauenburg
   (whose 8 drops were all same-nominal weight) correctly cleared.

2. **Two `_normalise_nominal` functions.** absorb imports the BARE normaliser
   from `lib.v2_seed_writer` (no synonym table) AND `match_pair` from the merge
   module (full synonym table). My first dry-run used the bare one → 18 drops
   including FALSE «1 Ducat» vs «1 Dukat», «2 Ducat» vs «1 Dobbelt Dukat»,
   «1 Gold Krone» vs «1 Guldkrone». Fix: import the merge module's normaliser
   as `_mg_normalise_nominal` for the identity check (consistency with the
   shipped discriminator). 18 → 12, all genuine.

### Decontamination is twin-independent (surgical-subtract)

Prior enrichment bakes every member's weight/sources onto the foundation
(`members[0]==fc`), so dropping a member isn't enough — its 0.44 g survives on
fc and re-pollutes. The over-merge purge resets to a twin; my hosts often have
none. Verified the no-twin hosts' accumulated data is PURELY member-derived
(km-56/km-582: 0 own-only sources; km-74: 4 ORPHAN sources backed by no current
member). So `_surgical_decontaminate` removes only values contributed by an
evicted member AND by NO remaining member — orphan + foundation-own data
preserved (§9a). Year left alone (overshoot is the §0-safe direction).

Re-home reuses the over-merge purge's `forced_evict_promote` set →
bulk-promote path (default mode `no_basic_peer_only` makes it fire). danish_realm:
12 evictions / 8 hosts → 10 standalone + 2 matched into correct existing finals
(`unified-dk-hede-c4h35` ½ Dukat, `unified-dk-hede-c4h99a` 1 Mark); 0 lost.
